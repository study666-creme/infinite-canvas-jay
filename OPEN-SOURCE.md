## AGPL-3.0 的实际含义

- 可以学习、修改、部署、收费托管和商业运营。
- 分发修改版本时，需要按 AGPL-3.0 提供对应源代码和许可证文本。
- 让用户通过网络使用修改版本时，需要向这些用户提供获取对应源代码的方式。
- 不能把 AGPL 派生代码重新标成闭源许可证，也不能增加与 AGPL 冲突的限制。
- 许可证不提供质量、适销性或特定用途保证。

具体义务应以 [LICENSE](LICENSE) 原文为准。闭源交付、企业内网、混合授权或插件边界等复杂场景应咨询专业法律人士。

## 产品 UI 与仓库归属

- 产品 UI 可以使用“卡藏提示词画布”品牌，不必长期展示上游仓库按钮。
- 当前仓库 README 与许可证应持续保留上游来源。
- 公网部署应在“关于”“开源说明”或其他易找到的位置提供当前部署对应的源码链接。
- 不要声称本仓库是完全从零开发，也不要删除第三方版权通知。

## 第三方代码与资产

- Three.js 使用 MIT 许可证，发布时需保留其版权和许可文本。
- npm 依赖、Codex SDK、图标和字体遵循各自许可证。
- 3D 模型、HDRI、贴图、图片、视频、音频和字体需要逐项确认授权；“可下载”不等于“可商用”。
- 用户导入的书籍、字幕、剧本和案例不会因为被本地资料库处理而改变版权状态。
- 自动生成的知识索引只应保存原创结构摘要、证据摘要和方法卡，不应发布受版权保护的完整原文或完整字幕。
